### Roadmap Phase 4 — HTTP request parser (`src/http/HttpRequest.*`) — PARTIAL
- **What:** `parseRequest()` takes the raw bytes up to `\r\n\r\n` and fills an `HttpRequest` struct: splits the request line into method / target / version, then parses each header line into a `std::map`.
- **Why these decisions:**
  - **Header names are lowercased** at parse time — HTTP header names are case-insensitive (`Host` == `host`), so normalising once means one clean `headers["content-length"]` lookup everywhere instead of case-juggling later.
  - **Query string is split off the path** (`/page?a=1` → path `/page`, query `a=1`). Without this, static file lookups break — the server would `stat("./www/page?a=1")` and 404. The raw query is kept for CGI's `QUERY_STRING` later.
  - **The path is percent-decoded** (`%20`→space, `%2e`→`.`) **inside the parser, before any validation.** This is a deliberate security ordering ("decode/canonicalise first, then validate"): the traversal check downstream must see the *real* path, or an attacker just encodes `..` as `%2e%2e` to slip past it. AI demonstrated both bugs live (a `?query` URL returning 404, and `%2e%2e` traversal not being blocked) before we fixed them.
- **Persistent per-connection read buffer:** because TCP is a byte stream, one `recv()` may deliver half a request. Each connection keeps a `std::string buffer` (in `ClientConnection`) that survives across `epoll_wait` iterations; we only parse once it contains `\r\n\r\n`. Verified by splitting a request across two writes with a `sleep` in between.
- **Deferred on purpose (see "Known gaps" below):** request **body** reading (`Content-Length`), **chunked** decoding, request-size **limits** (413/431/414), and a couple of status codes (505, missing-`Host` 400).

### Roadmap Phase 5 — Routing & responses (`src/http/Router.*`, `src/http/RequestHandler.*`) — DONE
- **Architecture split (important for the defense):** `EventLoop.cpp` is *only* socket/epoll plumbing — it never mentions HTTP status codes. All HTTP decisions live in `RequestHandler.cpp`'s `handleRequest()`, which returns a finished response *string* that the loop just ships. Easier to reason about and to test.
- **Location matching** (`matchLocation`) is **longest-prefix**, not first-match: for `/uploads/cat.png`, `/uploads` must win over `/`. First-match would let a `/` block (listed first) swallow everything.
- **`handleRequest` pipeline, in order:** traversal guard (`..`) → match location → `return` redirect (301/302 + `Location:`) → method allowed? (`405` + **`Allow:` header** listing the location's methods) → serve. The `Allow` list is built from the *same* `loc.methods` used for the check, so it can never disagree with the decision.
- **Error pages funnel through one `errorPage()` function** with a two-step fallback: serve the configured `error_page` file (resolved against the location's `root`) if it reads, else a generated HTML stub — guaranteeing a 4xx/5xx is *never* empty (a subject requirement). Limitation: when there's no matched location (a traversal 403 / no-catch-all 404) there's no `root` to resolve against, so those use the stub.

### Roadmap Phase 6 — GET static files (`RequestHandler.cpp`) — DONE
- **Path mapping:** filesystem path = `location.root + request.path` (e.g. `./www` + `/uploads/hello.txt`).
- **Security:** any decoded path containing `..` → 403 (tested with plain, fully-encoded, and mixed-encoding traversal attempts — all blocked).
- **`stat()` dispatch:** regular file → serve it; directory → prefer the `index` file if it exists, else an **autoindex** listing if `autoindex on`, else 403; missing → 404.
- **Autoindex** builds an HTML directory listing with `opendir`/`readdir`/`closedir`; links are written **absolute** (`/uploads/notes.txt`) so they resolve correctly whether or not the request had a trailing slash.
- **MIME types** from file extension (html/css/js/png/…); unknown → `application/octet-stream`.
- **Response writing via `EPOLLOUT` (this replaced the Phase-2 inline `send`):** once the response string is built we store it in the connection's `writeBuffer` and flip the fd from `EPOLLIN` to `EPOLLOUT` with `epoll_ctl(MOD)`; we only `send()` when epoll reports the socket writable (subject rule: never write without readiness), and on a **partial send** we erase the bytes that went out and keep the rest for the next `EPOLLOUT`. Proven by downloading a 2 MB file intact — a single `send()` can't hold that, so it exercised the partial-write loop.

## 2026-07-18 (session 2) — Roadmap Phase 4 finish: request-body reading + `client_max_body_size`

- **Deadline update:** the self-imposed deadline was extended to **Monday 2026-07-20** (weekend added as working days), so the pace eased slightly — still teach-first, but scaffolding help stays on the table.
- **The core new idea — a two-stage completeness check** in `EventLoop.cpp`. The old loop had a *one-stage* rule ("saw `\r\n\r\n` → request done"), correct only for bodiless GETs. Now:
  - **Stage 1:** is the whole header block here? (`buffer.find("\r\n\r\n")`) — if not, wait for the next `EPOLLIN`.
  - **Stage 2:** parse the headers, read `Content-Length`, and only dispatch once `buffer.size() >= headerEnd + contentLength`; otherwise keep reading. `headerEnd = pos + 4` (the 4 separator bytes `\r\n\r\n`) is the first body byte.
- **`client_max_body_size` → `413 Payload Too Large`**, enforced **in the event loop, before routing** — the whole point is to refuse an oversized upload *up front* from the `Content-Length` header, without ever buffering the body into memory. AI made the case that rejecting as early as you have enough info to reject is both a memory-safety and an anti-DoS decision.
- **Small refactor:** extracted a `flipToWrite(epfd, fd)` helper (the `EPOLLOUT` `MOD` dance now happens in 4 places), and added a `std::string body` field to `HttpRequest`.
- **What AI did vs. what I wrote:** I typed the `HttpRequest.body` field myself; when the branching logic in the read path got hard to track mentally, AI wrote the full revised `EventLoop.cpp` read branch (I read the annotated version rather than diffing it in my head). I ran all tests.
- **Verified with curl:** a small `--data` POST returns (doesn't hang) → body consumed whole; an over-limit body returns `413`; GET still serves normally. Confirmed the `413` fires *before* the `405` method check, proving the ordering.
- **Oral-defense drill** on the new code. Two answers worth keeping: (1) an incomplete body (client promises 5000, sends 3000, goes silent) currently **hangs the connection forever** — this is *why the subject mandates timeouts*, a Phase 10 item; (2) the `<` (strict) in the Stage-2 wait condition is what makes an exactly-complete body count as *done* rather than waiting one read too long.
- **Still deferred (unchanged):** chunked decoding (→ Phase 9/CGI), `411 Length Required` for a POST with no length, `400` on non-numeric `Content-Length`, `414`/`431`. Known `TODO(phase10)`: the read buffer isn't shrunk after dispatch — harmless while responses send `Connection: close`, a real bug once keep-alive lands.

## 2026-07-18 (session 2 cont.) — Roadmap Phase 7: POST uploads (raw + multipart)

- **Stage 1 — raw-body upload** (`serveUpload` in `RequestHandler.cpp`): writes the already-buffered `req.body` to `upload_store + "/" + basename(URL path)`, returns `201 Created` + `Location`. Filename comes from the **last path segment**. Added a `writeFile` helper — the mirror of Phase 6's `readFile`, and the same "**regular files are exempt from epoll**" point applies: a plain synchronous `write()` loop, no epoll registration (only sockets and CGI pipes go in epoll). Failures decided from the return value only (no `errno`).
- **Stage 2 — multipart/form-data** (`parseMultipart`): a browser `<form>` doesn't send raw bytes, it sends a *named-parts envelope* so a form can carry multiple labelled fields with metadata (the original filename lives in the part's `Content-Disposition`, not the URL). Parser: pull `boundary=` from `Content-Type`, find the part with `filename="..."`, and slice the bytes between that part's blank line (`\r\n\r\n`) and the next boundary (`\r\n--BOUNDARY`). Those exact `\r\n` offsets are what keep binary uploads byte-perfect. `serveUpload` now branches on `Content-Type`: multipart → `parseMultipart`, else → raw body.
- **Security (defense point):** the multipart filename is **attacker-controlled**, so it's stripped to basename before use — same guard as the URL path, plus the existing `..` traversal check in `handleRequest`. Two independent layers keep every write inside `upload_store`.
- **Status codes wired:** `201` (+`Location`), `403` (uploads disabled / no `upload_store`), `400` (malformed multipart / no filename), `500` (write failed). Added a full **HTTP status-code reference table** to `README.md` (implemented ✅ vs planned ⏳) and filled in the README **Testing** section with the curl upload commands.
- **Verified:** raw upload (`curl --data-binary`) and multipart upload (`curl -F`) both round-trip a large binary (the `webserv` executable itself) with `cmp … && echo IDENTICAL` — byte-exact. Browser demo page added at `www/upload.html` for the evaluation's "must work from a real browser" requirement.

## 2026-07-19 — Roadmap Phase 9: CGI (+ deferred chunked decoding)

- **The architectural shift (the big teaching moment):** every earlier phase was *synchronous* — `handleRequest()` builds the whole response and returns a string the loop ships. CGI breaks that: the response doesn't exist until a **child process** runs (possibly for seconds), and we may never block. So a CGI request becomes a small **state machine attached to the client** that advances across many `epoll_wait` iterations. AI walked through why the naive "`read()` the script's output in a loop" is an automatic grade-0 (one slow script freezes every other client), and why the fix is to treat the CGI pipe fds exactly like sockets — non-blocking fds registered in the *same* epoll instance.
- **`CgiHandler.cpp` (`startCgi`) — the fork/exec plumbing:** two `pipe()`s (server→child stdin, child stdout→server), `fork()`, and the fd dance — **each side closes the two ends it doesn't use** (forgetting even one means EOF never arrives and the request hangs). Child: `dup2` the pipe ends onto fds 0/1, `chdir` to the script's directory (subject requires correct relative paths), `execve(interpreter, {interpreter, script, NULL}, envp)`. Defense points AI drilled: `_exit(1)` (not `exit`) after a failed `execve` so the child doesn't flush the parent's duplicated stdio buffers; and only the *parent's* two pipe ends are set `O_NONBLOCK` — the child's stdin/stdout stay blocking on purpose (a separate process is allowed to block on its own I/O). `buildCgiEnv` fills the RFC 3875 meta-variables plus every request header as `HTTP_*`.
- **Design split (defense):** `CgiHandler` owns the messy fork/exec and hands back `{pid, inFd, outFd}` but **never touches epoll** — all `epoll_ctl` stays in `EventLoop.cpp`, preserving the "one epoll instance, one place that mutates it" invariant an evaluator checks.
- **EventLoop integration:** `ClientConnection` gained a `ConnState` enum (`READING`/`CGI_RUNNING`/`WRITING`) + CGI fields, and a `pipeOwner` map (pipe fd → owning client fd) so a ready pipe fd can be routed back to its client — because all `epoll_wait` gives us is `data.fd`. The dispatch became **three-way** (listener / pipe / client). On launch we `stat` the script (404 if missing, no pointless fork), fork, set the **client fd's epoll interest to 0** (it goes idle — it's neither reading nor writing while the child runs), register `outFd` for `EPOLLIN` and (only if there's a body) `inFd` for `EPOLLOUT`. When the body is empty we `close(inFd)` immediately so the child sees EOF on stdin at once. On the output pipe's EOF (`read()==0`) we `waitpid` (reap), `buildCgiResponse`, and flip the client to `EPOLLOUT` — rejoining the existing send path.
- **The #1 bug AI diagnosed — a level-triggered `EPOLLHUP` busy-spin.** First test hung: `curl` got 0 bytes, the server sat at **91% CPU**, and the python child was a `<defunct>` zombie. AI read those three symptoms together: after the script's output is drained and its write-end closes, Linux stops reporting `EPOLLIN` and reports **`EPOLLHUP`** instead — which epoll delivers whether or not you asked for it. My handler only acted on `EPOLLIN`, so the HUP was never consumed, the fd never closed, and level-triggered epoll re-reported it *every* iteration forever (hence the spin and the never-reaped child). Fix: gate the read/write branches on `(EPOLLIN|EPOLLHUP|EPOLLERR)` / `(EPOLLOUT|EPOLLHUP|EPOLLERR)` and let the `read`/`write` return value drive the decision. This is the canonical CGI-pipe gotcha and a strong oral-defense story.
- **CGI timeout → `504` (the "requests never hang" guarantee):** changed `epoll_wait`'s timeout from `-1` to `1000 ms`, and each iteration `reapTimedOutCgi()` scans running CGI jobs for `now - cgiStart >= CGI_TIMEOUT` (5s), `kill(SIGKILL)` + `waitpid` to reap, tears down both pipes, and answers `504`. Defense nuance: a **blocking** `waitpid` is acceptable *here* only because `SIGKILL` guarantees the child dies promptly — it's not waiting on arbitrary child work.
- **Chunked decoding (the Phase-4 debt, mandatory for CGI):** `decodeChunked()` in `HttpRequest.cpp` returns an enum (`CHUNK_INCOMPLETE` = wait for more bytes / `CHUNK_DONE` / `CHUNK_ERROR` = 400), looping `<hex-size>\r\n<data>\r\n` until the `0`-chunk + terminating CRLF; the hex size parser reuses the existing `isHexDigit`/`hexValue` helpers and strips any `;ext`. `EventLoop` Stage 2 now branches on `Transfer-Encoding: chunked` vs `Content-Length`; **both converge to the same dispatch**, so the CGI (and every handler) only ever sees a plain, un-chunked body. `client_max_body_size` is re-checked on the decoded size.
- **Test scripts added** (`www/cgi-bin/`): `hello.py`, `echo_post.py` (reads stdin, echoes it), `env.py` (dumps env vars — debug), `slow.py` (`sleep(9999)` — must trigger 504 while others keep working).
- **Verified end-to-end:** GET `hello.py` → 200 + HTML; POST → `echo_post.py` echoes the stdin body; **chunked** POST arrives byte-identical to the `Content-Length` form; malformed chunk size → 400; missing script → 404; `slow.py` → **504 after ~5s** *while* a concurrent `hello.py` returns 200 in 0.01s (proves one stuck CGI doesn't block the loop); no leftover children, server CPU 0%. All under `-Wall -Wextra -Werror -std=c++98`.
- **What AI did vs. what we wrote:** AI taught the pipe↔epoll model and the two-pipe fork/exec design as Q&A first, then scaffolded `CgiHandler`, the state-machine fields, and the event-loop wiring (which we applied and tested); crucially it *diagnosed* the `EPOLLHUP` spin from the CPU/zombie/0-byte symptoms and guided the fix, and designed the 504 reaper and the chunked decoder. We ran every test.

## 2026-07-19 — Roadmap Phase 10: multi-server & robustness

- **`SIGPIPE` ignore (the one-liner that saves the server):** `signal(SIGPIPE, SIG_IGN)` at the top of `runEventLoop`. AI's framing: a client vanishing mid-`send()` would otherwise deliver `SIGPIPE` whose *default action is to terminate the process* — the whole server, not the connection. We don't need a handler because we already detect the dead peer from `send()`'s `-1` return (consistent with the project's return-value-only rule); a signal *handler* would just drag in async-signal-safety constraints for no gain.
- **Keep-alive — the two-sided design AI drilled:** what we *tell* the client (the `Connection` header) and what we *do* with the fd (reuse vs `close`) must agree, or a client reuses a dead socket. Rule: HTTP/1.1 is persistent unless `Connection: close`; HTTP/1.0 is close unless `Connection: keep-alive` (`shouldKeepAlive(req)`). The builders emit a safe `Connection: close` default; `applyKeepAlive()` upgrades that single header line in place only when the connection qualifies. **Scoping decision (defensible):** only successful `handleRequest` responses reuse — **every error and every CGI response closes**, because after a parse error the request stream is in an unknown state, and CGI output isn't always length-delimited (and keep-alive *requires* a body delimiter). Committed only at the final dispatch via `keepAlive = reusable && shouldKeepAlive(req)`, where `reusable` is set true solely in the non-chunked body branch — **chunked forces close** because `decodeChunked` reports the decoded body but not how many *raw* bytes it consumed, so we can't erase the request precisely.
- **The `EPOLLOUT` completion block became three-way:** `send() <= 0` → drop; fully sent **and** keep-alive → **reset** (`buffer.erase(0, consumed)`, back to `EPOLLIN`, state `READING`); else `close`. AI's precision point: reset uses `erase(0, consumed)`, **not** `buffer.clear()` — a client may have **pipelined** request #2 into the buffer before reading response #1, and `clear()` would silently drop it and hang the client. Honest limitation noted for defense: because epoll is level-triggered on the socket, a *pipelined* follow-up already sitting in our buffer won't re-fire `EPOLLIN` until new bytes arrive — acceptable since mainstream browsers don't pipeline (they open parallel connections).
- **Idle client timeouts (why keep-alive made this urgent):** the moment connections stop closing after one response, an abandoned or slow-loris client holds an fd forever. `time_t lastActive` on `ClientConnection` (stamped at `accept`, on every `recv`, and on every `send > 0` — "any progress resets the clock"); `reapIdleClients()` closes connections idle past `CLIENT_TIMEOUT` (30s). **It deliberately skips `CGI_RUNNING`** — those look idle (client fd interest is 0 while the child runs) but are legitimately busy, and closing one would orphan the child (zombie) and leak its pipe fds; `reapTimedOutCgi` owns their lifetime. AI flagged the C++98 gotcha: `clients.erase(it++)` (post-increment first) is required because `erase` invalidates the iterator — `erase(it); ++it;` is undefined behavior.
- **Multi-server on different ports:** `config/two_sites.conf` runs `:8080` (root `./www`) and `:8081` (root `./www2`) simultaneously through the one epoll loop. Defense point AI surfaced: the `listenFdToServer` map records which listening socket `accept()`ed a connection, pinning it to that `serverIndex` for its whole life, so per-port content is automatic. **Known boundary (stated honestly, beyond mandatory):** one listening socket per `server` block means same-port name-based virtual hosts (`server_name`/`Host`) aren't supported — a second `bind()` to the same port fails `EADDRINUSE`.
- **Crash-proofing sweep (subject: "never crash even on out-of-memory"):** AI audited for `throw` sites, raw `new`/`delete` (none), and `.at()` (none). Findings: (1) `epoll_wait` returning `-1` threw unconditionally — now `continue`s on `EINTR` (a signal interrupting the wait is benign) and only throws on a genuine, unrecoverable epoll failure; checking `errno` here is allowed since it's not a socket read/write decision. (2) The whole per-event body was unguarded — a `std::bad_alloc` from any string/container growth would unwind straight through `runEventLoop` → `main` → **the entire server exits because one connection hit OOM**. Fixed by wrapping the per-event body (and the reapers) in `try/catch(std::exception)` → log to `cerr` + continue, isolating a failure to the single connection that caused it (a wedged one is then reaped by the idle timeout — the pieces cover each other). The startup throws in `ConfigParser`/`Listener`/epoll-setup are left as-is: they're caught by `main` and exit(1), which is *correct* — a bad config or unbindable port should refuse to start.
- **Verified:** `curl` reuses one connection across two requests on HTTP/1.1 and honors an explicit `Connection: close`; HTTP/1.0 still serves (and closes); an abandoned raw socket is reaped after the timeout while normal requests are unaffected; `:8080` and `:8081` serve different content at once; 200 concurrent requests + garbage input leave the server alive. All under `-Wall -Wextra -Werror -std=c++98`.
- **What AI did vs. what we wrote:** AI taught the keep-alive rule, the two-sided header/fd agreement, and the timeout/crash-proofing rationale as Q&A first, then specified each small edit (helper signatures, the three-way `EPOLLOUT` block, the reaper, the `try/catch` wrap); I typed and adapted every edit and ran all the tests. AI also drove the milestone quiz (connection-lifecycle questions) for oral-defense prep.

## 2026-07-19 — Roadmap Phase 11: testing & hardening

- **Parser hardening — the hostile-input status codes.** AI's design call: change `parseRequest` from returning `bool` to returning an **HTTP status code** (`0` = success), so it reports precise codes instead of a blanket 400 in two scattered places. Added: `414` (request line > 8 KB — bounds memory rather than buffering a giant URI), `505` (version not `HTTP/1.1`/`HTTP/1.0`), and in the event loop `431` (header block passes a 16 KB cap with no end-of-headers — the answer to "why is unbounded `recv`+append dangerous": a client that never sends the blank line would grow the buffer forever), non-numeric `Content-Length` → `400`, and `411` (a POST with neither `Content-Length` nor chunked framing — the body has no delimiter, so we refuse it). A `reasonPhrase(int)` helper maps each code to its text.
- **A real fd leak `--track-fds` caught.** `main` opens the config with an `std::ifstream`, but because `runEventLoop` never returns, that stream is **never destroyed** — so the server holds the config fd open for its entire lifetime. Valgrind reported it as "open at exit". Fix: `file.close()` right after slurping the contents. AI's framing of *why it hid*: it's not a "leak" in the `new`/`delete` sense (the object is a live local), which is exactly why only an fd-tracking tool surfaces it — a good demonstration of *why* `--track-fds=yes` is worth running, not just `--leak-check`.
- **The Python torture harness (`tests/torture.py`).** AI wrote it as the phase's verification tool. Design points it made: it uses **raw sockets, not `http.client`**, specifically so it can send *illegal* bytes (bad version, non-numeric length, `\x00\x01\x02` garbage) that a well-behaved HTTP client library would refuse to construct. It answers "why multithreaded?" — 50 threads hammering shared server state is what flushes out concurrency bugs (fd races, one-slow-client-blocks-everyone, keep-alive buffer corruption) that a sequential loop structurally cannot reach. Each check asserts a specific status code; the run ends by asserting the server still answers `200`.

## 2026-07-19 — Passing the official 42 `cgi_tester` (`./tester`)

The single most productive debugging session: running the provided `./tester` binary end-to-end surfaced a string of bugs the happy-path tests had missed. Each failure was diagnosed by reproducing the *exact* request the tester sends, then reading the symptom. Two of the finds were genuine **O(n²)** bugs that only a large, streaming, concurrent client triggers.

- **Setup (not a bug, a gotcha).** `./tester` is a *client* — it never launches the server. The first "connection refused" simply meant no server on `:8080`; you run `./webserv config/tester.conf` in one terminal and `./tester` in another. AI built `config/tester.conf` (the four required blocks: `/` GET-only, `.bla` → CGI, `/post_body` capped, `/directory` with `index youpi.bad_extension`) and the `YoupiBanane/` fixture. Key detail: our path mapping is `root + request_path` (not nginx `alias`), so the `/directory/*` files had to physically live under `YoupiBanane/directory/`.
- **`PATH_INFO incorrect` → the CGI needs `REQUEST_URI`.** The `.bla` POST returned a 500 whose body (`PATH_INFO incorrect`, `Content-Type: …charset=utf-8`) was cgi_tester's *own* output — so the CGI ran fine and was rejecting our environment. Hours of bisecting the (stripped-of-nothing, but opaque) Go binary by replaying captured env var sets showed the real requirement: cgi_tester cross-checks `REQUEST_URI` against `SCRIPT_NAME`/`PATH_INFO`, and **we never set `REQUEST_URI` at all**. Fix in `buildCgiEnv`: emit `REQUEST_URI` (path + query), with `REQUEST_URI == SCRIPT_NAME == PATH_INFO == req.path`. A dead end worth recording: with `SCRIPT_NAME` present, *no* `PATH_INFO` value works until `REQUEST_URI` is also there.
- **HEAD — responses must never carry a body.** HEAD on the GET-only `/` first returned `405` *with* an HTML body; the Go client errored `Unsolicited response received on idle HTTP channel` because a leftover body desyncs a keep-alive connection. Making HEAD serve like GET (200) was also wrong — the tester answered `bad status code`. The real rule, read off the two error messages: HEAD is checked as its own method (so GET-only ⇒ `405`), served like GET only where HEAD is allowed, and **every** HEAD response (including the `405`) is passed through a `stripBody()` that cuts everything after the blank line while leaving `Content-Length` intact.
- **A directory with no index ⇒ 404, not 403.** The tester's `GET /directory/Yeah` (a real subdir lacking the configured `youpi.bad_extension` index) expects `404` — it treats "directory with no servable index" the same as a nonexistent path. Changed `serveGet`'s no-index/no-autoindex branch from `403` to `404` (and set `autoindex off` on `/directory`). Defensible either way; we matched the tester (nginx would say 403).
- **Per-location `client_max_body_size` (a genuine feature).** The tester POSTs **100 MB** to `/directory/youpi.bla` and expects it *accepted* — the `100`-byte cap belongs only to `/post_body`. Our `client_max_body_size` was server-wide, so it wrongly rejected the CGI upload (`413`) and the tester hung. Added the directive to `LocationConfig` (with a `has_max_body` override flag), parsed it inside `location` blocks, and had the event loop resolve the effective cap per longest-prefix match before reading the body.
- **The first O(n²): the chunked decoder.** After the cap fix the 100 MB POST *still* hung at ~92 % CPU. The tell: **curl passed, the Go tester didn't** — because the tester streams the body `Transfer-Encoding: chunked` while curl sent `Content-Length`. The old decoder ran `decodeChunked(buffer.substr(headerEnd))` on *every* `recv`: a full copy of the body-so-far plus a re-walk of every chunk from byte zero, ~24 000 times. Rewrote it as **resumable** — a per-connection `chunkPos` offset and a `chunkBody` accumulator, so each arriving packet is parsed exactly once. 100 MB went from "never finishes" to **1.7 s**. The cap is now also enforced *while streaming*, not only at the end.
- **A missing `.bla` file must still reach the CGI.** `POST /directory/youpla.bla` (note the typo — the file doesn't exist) expects `200`: cgi_tester ignores file existence and answers anyway. Our pre-`fork` `stat` → `404` was wrong for CGI. Fix: don't refuse the launch on a missing *file* (only a *directory* still 404s early); record `cgiScriptMissing`, and at CGI EOF answer `404` **only if** the child produced no output at all — so a missing Python script (empty stdout, error on stderr) still yields `404`, while cgi_tester's own `200` wins.
- **The second O(n²): the send path — the best find of the day.** The final failure was `multiple workers(20) … Post … size 100000000` — twenty concurrent 100 MB CGI posts, one returning non-200. The `EPOLLOUT` handler did `writeBuffer.erase(0, s)` after every partial `send`: erasing the *front* of a 100 MB response `memmove`s the entire remaining tail on **every** send, and under concurrency the kernel hands back small slices, so a single response cost hundreds of GB of `memmove`. Replaced with a `writeOff` offset that just advances — the exact pattern the CGI-feed path (`cgiInOff`) already used. Same moral as the chunked bug: *never redo work on bytes you've already processed.*
- **Concurrency tuning that shipped with it:** `CGI_TIMEOUT` 5 s → 30 s (five seconds was killing healthy jobs that share one event loop → spurious `504`); `BUF_SIZE` 4 KB → 64 KB (16× fewer syscalls per MB); raw chunked bytes reclaimed from the buffer as they decode; the body handed to the CGI by `swap` instead of a 100 MB copy; CGI in/out buffers freed the instant they're consumed.
- **Verified:** twenty concurrent 100 MB chunked CGI posts → **twenty `200`s in ~10 s**, server alive; the earlier `tests/torture.py` suite still **all-PASS** after every change; `./tester` runs to completion.
- **What AI did vs. what we wrote:** AI drove the diagnosis of each tester failure — reproducing the precise failing request, reading the symptom (413 vs hang vs 92 % CPU vs framing error), and isolating the cause (e.g. bisecting the Go binary's env-var checks, spotting that curl-passes-but-Go-hangs implicated chunked). It applied the fixes under time pressure and re-verified each at the tester's scale. I re-ran `./tester` after each fix and reported the next failure. The two O(n²) diagnoses and the `REQUEST_URI` discovery were the high-value moments.
- **Milestone: the official `cgi_tester` passes.** Two O(n²) bugs, per-location body limits, HEAD, and CGI env correctness were all shaken out by it — exactly why the subject says to run the provided tester.
